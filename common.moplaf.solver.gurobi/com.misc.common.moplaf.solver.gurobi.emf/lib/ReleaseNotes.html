Obtaining Your License

Gurobi Optimizer 5.5 uses the same licenses as version 5.0, so if you already have a Gurobi 5 license, there's no need to obtain a new one. You may need to copy your existing license to a new location, though. Consult the Quick Start Guide for information on where to store your license file.

If you don't already have a Gurobi 5 license, you will need to visit the Licenses page on the Gurobi website. If you are a commercial user under maintenance, you should see your Gurobi 5 license under Current. Otherwise, you can request a free trial or free academic license. Once you have created a license on the Gurobi web site, you will need to follow the instructions for obtaining and installing a license in the Quick Start Guide.

Note that our new Gurobi Compute Server feature requires a special license type, so this feature won't be enabled with an existing Gurobi 5 licenses.

Release Notes for Version 5.5

Gurobi 5.5 contains a number of new features and capabilities. Some users of previous versions may need to make minor changes to their programs to use the new version. The new capabilities are described here.

  • Gurobi Compute Server: This new feature allows you to seamlessly offload optimization tasks onto one or more compute servers. The feature is mostly transparent, but it does expose few new functions, parameters, and error codes. Consult the Compute Server section of the Reference Manual for details.
  • Automatic parameter tuning tool: This new tool automatically considers multiple different parameter settings to find a set that improve performance on a particular model. It is available in a command-line tool (grbtune), and through our programming language API's. The tuning tool introduces several new tuning-specific parameters:
    • TuneTimeLimit: Sets the time limit for tuning.
    • TuneOutput: Controls the amount of output produced by the tuning tool.
    • TuneResults: Controls the number of improving parameter sets that are retained.
    • TuneTrials: Controls the number of trials that are run for each candidate parameter setting.
    It also introduces one new tuning-specific attribute:
    • TuneResultCount: Indicates the number of improving parameter sets that were retained.
    Consult the Parameter Tuning Tool section of the Reference Manual for details.
  • Concurrent MIP Optimizer: This new feature launches multiple, independent solves on the same MIP model, using different settings for each. The solve returns when the first one finishes. This approach allows you to exploit multiple cores to explore a diverse set of search strategies. This feature is enabled using the new ConcurrentMIP parameter. Consult the Concurrent Optimizer section of the Reference Manual for details.
  • New NumericFocus parameter: This new parameter allows you to indicate that a model is likely to experience numerical trouble, which then causes our internal algorithms to favor numerical robustness over speed.
  • New LazyConstraints parameter: This new parameter provides a more direct way of indicating that your program uses lazy constraints. Use this instead of the previous approach of setting the DualReductions parameter.
  • New MIPGap attribute: This new attribute allows you to query the current MIP optimality gap directly, rather than having to compute it from ObjVal and ObjBound.

Code Conversion from Version 5.1

As mentioned earlier, minor changes may be required for some users of Gurobi 5.1:
  • Lazy constraints: If your program uses lazy constraints, you will need to set the new LazyConstraints parameter (and you no longer need to set the DualReductions parameter).
  • Floating licenses: Users of floating or single-use licenses may notice that the name of the token server has changed. It was previously called grbs on Windows and grbd on Linux/Mac. The new name is grb_ts.
